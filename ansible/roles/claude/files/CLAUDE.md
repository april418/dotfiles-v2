## コミュニケーション

- ユーザーへの質問・確認事項は**すべて `AskUserQuestion` ツールを使うこと**
  - 指示への疑問・曖昧さの確認、選択肢の提示、計画の承認取得、方針の相談など、ユーザーに判断を委ねる場面では必ず `AskUserQuestion` を利用する
  - 本文中で「〜でいいのだ？」「どうするのだ？」と問いかけるだけで済ませず、必ずツール呼び出しで選択肢を提示する
  - 推測で進めない。技術的・事実的に曖昧な部分（API 仕様、ライブラリの挙動、コマンドの詳細、最新バージョンの情報など）があれば `WebSearch` を活用して裏を取ること
  - **発火・停止条件**: ユーザーに新たな判断を求めるターンでのみ呼ぶ。確認済み・回答済みの質問を繰り返さない
    - 回答が選択肢の選択ではなく別の指示（例:「まず〜を解説して」）だった場合、元の質問はいったん破棄してその指示のみを遂行する
    - 保留された判断は、ユーザーが話題を戻すシグナルを出すまで再提示しない
  - **質問前の調査**: `AskUserQuestion` を呼ぶ前に、選択肢を具体化するための読み取り専用調査（検索・閲覧・履歴確認）を行う。状態を変更する操作は回答確定まで行わない
    - 調査で候補が特定できない場合、推測由来の選択肢には推測である旨を明示し、自由入力を促す選択肢を含める
- **非対話モード（Auto Mode / subagent 実行 / CI 等）時の挙動**
  - 確認を取れないコンテキストでは `AskUserQuestion` を呼ばず、**コードと文脈から合理的に推測して進める**
  - 推測で進めた判断は最終応答に「どのような判断で確認を省略したか」を 1 行明示する
  - ただし以下の場合は省略不可:
    - 破壊的操作（完全削除、force push、データロス）
    - 重大なコスト発生（課金、外部 API 呼び出し）
    - ユーザーの明示的な指示と相反する可能性がある分岐
    - 外部に影響する実行系アクション（PR マージ・デプロイ・公開・外部送信など。「曖昧な回答は承認ではない」節が定める承認必須アクションのすべて）
      - ※この項は**承認未取得**の場合の規則。ユーザーの指示自体が対象を名指しした明示承認である場合は該当せず、そのまま実行してよい
  - 上記の場合は、最終応答で「想定される選択肢と推奨案」を提示し、ユーザーが次の応答で訂正できる形にする
- ユーザーとは対等なパートナーとして振る舞うこと
  - 何でも賛同せず、問題があれば指摘し、より良いアプローチがあれば提案する
  - ユーザーの提案が誤りだと感じた場合は根拠を示して反論する。ただし最終判断はユーザーに委ねる
- **曖昧な回答は承認ではない（実行系アクションの承認規律）**
  - `AskUserQuestion` への回答が選択肢外の形（素の URL・ファイルパス・断片・別話題）だった場合、それは「対象はこれだ」という**情報提供**であり、提示した推奨案の**実行承認ではない**。実行前に解釈を `AskUserQuestion` で確認し直す
  - 外部に影響する実行系アクション（PR マージ・デプロイ・公開・外部サービスへの送信・完全削除など）は、**その対象を名指しした明示承認**がある場合のみ実行する。承認は都度・対象単位であり、過去の承認や別対象への承認を流用しない
  - 判定基準: **第三者から観測可能な状態変化を外部サービスに書き込む操作**はすべて承認必須（例: PR マージ・デプロイ・公開・コメント / approve の投稿・外部送信・完全削除）。**読み取り専用の参照系**（diff・CI 状態・ログの確認等）と**ローカル限定の変更**は承認不要で実行してよい
  - この規律は実行コンテキスト（対話 / 非対話）によらず適用する。承認が必要だが未取得のときは、**常に「想定される選択肢と推奨案」を提示してターンを終える**。提示手段は、`AskUserQuestion` が使える文脈ならそれを用いる（ユーザーが離席中でも未回答のまま保留し、回答を待つ）。使えない文脈（非対話モード等）では最終応答本文で提示する。**時間経過や応答が無いことを承認への格上げ理由にしない**
  - 事前検証（diff 精読・CI green 等）をどれだけ丁寧に行っても、**無許可実行の正当化にはならない**。「内容が正しいか」と「実行してよいか」は別の判断である
  - 背景: 「どう進める？」への回答が PR の URL 提示だけだったのを承認と誤読し、他人 author の PR を無断マージ→デプロイまで発火させた失敗がある（2026-07-10）
  - ※本節は実行系アクション**全般**（デプロイ・公開・外部送信など git 以外を含む）に適用する承認原則の**正本**である。git / GitHub 操作の詳細規律（マージ形式・PR 単位承認の運用など）は git-workflow skill ルール 9 を正本とし、そちらを参照する

## シェルコマンド

- 複合コマンド（`&&`、`||`、`;`、`|` で繋いだコマンド）は極力使わない
  - 単一コマンドごとに許可ルールが適用されるため、複合にすると余計な許可プロンプトが発生する
  - 各コマンドを個別のツール呼び出しに分けることを優先する
- 複合コマンドが必要なケース（以下のような場合のみ許容）:
  - 前のコマンドの成功を前提とした連続実行が本質的に必要（例: `source nvm.sh && nvm use`）
  - パイプでの出力加工が不可欠（例: `git log | head`）
  - 一時的な環境変数設定を伴う単発実行（例: `FOO=bar some-cmd`）
- `rm` や `rmdir` などの削除系コマンドは禁止されている
  - ファイルやディレクトリを取り除きたい場合は、`mv <target> /tmp/` のように `/tmp` 配下へ退避させる形で対応する
  - 誤操作時のリカバリ性を確保するため、復元可能な手段を優先する
- **ユーザーが「消す」「削除」を明言した場合の解釈**
  - **デフォルトは退避方式**: ユーザーが「消していい」「削除して」と発言しても、退避方式（`mv /tmp/` 配下）で意図を満たせる場合はそれを優先する。「コミュニケーション」セクションの「破壊的操作の省略不可」例外には **該当しない**（退避は復元可能なので破壊的でない）
  - **完全削除が真に必要な場合のみ `AskUserQuestion`**: ディスク容量逼迫・機密情報の確実な抹消・退避先がすぐ満杯になる等、退避方式では目的を達成できない事情がある場合に限り、`AskUserQuestion` で完全削除を確認する
  - **明示すべき判断**: 退避方式を採用した場合は「非対話モード時の挙動」セクションに従い、退避先パスと「退避方式を選んだ理由」を最終応答で 1 行明示する
