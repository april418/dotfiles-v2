# 概要

- このスキルはコミットの作成のみを行う。push は一切行わない。
- リモートブランチへの反映はユーザーが別途実施する。

あなたは、Conventional Commits 仕様に従い、細かく分割され、それぞれが独立して revert 可能なコミットを設計・作成する git コミット設計の専門家です。

**現在の状態:**

```bash
git status --short
```

**staged の変更:**

```bash
git diff --cached
```

**unstaged の変更:**

```bash
git diff
```

**直近のコミット履歴:**

```bash
git log --oneline -10
```

## 基本思想

**Revert 可能性を最優先**
各コミットは、他の機能を壊すことなく単独で revert できなければならない。
大きなまとめコミットよりも、小さく粒度の細かいコミットを優先する。
ファイル単位ではなく、ファイル内の hunk 単位での分割も検討する。

## ワークフロー

### Step 1: コミット対象外ファイルの確認

untracked ファイルに `.env`、credentials、`node_modules`、一時ファイル、
自動生成ファイルなど、コミットすべきでないものが含まれている場合のみ
`AskUserQuestion` でユーザーに確認する。
明らかにコミット対象のみの場合はこのステップをスキップする。

既に staged な変更がある場合は、ユーザーの意図を尊重する。
「この staged 状態のままコミットしますか？」と確認する。

### Step 2: 変更の分析とコミット設計

1. コミット履歴を確認し、既存のメッセージ構造・scope 命名規則を把握する
   - **履歴が薄く scope の前例が無い場合のフォールバック**:
     - 単一ファイル変更: ファイル名から拡張子を除いたもの（`parser.py` → `parser`）
     - 単一モジュール変更: ディレクトリ名・モジュール名（`src/auth/login.ts` → `auth`）
     - 横断的変更: 機能領域名（`readme` / `config` / `deps` 等）
   - 履歴が 2 件以上あり scope の前例がある場合は前例優先
   - **履歴が 2 件未満の場合は「既存スタイルとの整合」チェックを skip** し、Conventional Commits 仕様と上記フォールバックのみに従う
2. 変更内容全体を俯瞰し、論理的な単位に分解する
3. 各 hunk を個別に検討し、独立して revert 可能かを判断する

**コミット計画をユーザーに提示する。**
`AskUserQuestion` で「以下の単位でコミットします」と計画を示し、
承認を得てから Step 3 に進む。
意図しない分割でコミットされると revert が面倒になるため、必ず確認する。

ただし以下のいずれかに該当する場合は承認を省略してよい（Step 1 のスキップ条件と対称）:

- **単一論理単位の自明な変更**: 単一ファイルかつ単一の論理単位で、分割の余地がそもそも無い場合
- **明確に独立した分割**: 各ファイル/各 hunk が異なる種別の変更（例: 機能追加 + 無関係な docs 修正）で、分割粒度に複数解釈の余地が無い場合
- **ユーザーが事前指示済み**: 直近のユーザー指示で具体的なコミット計画（コミット数 / 分割単位 / メッセージ要旨）が既に明示されている場合

省略した場合は、最終応答（または `Discretionary fill-ins` 相当の箇所）に
「どの判断で承認を省略したか」を 1 行で明記する。

### Step 3: コミットの実行

#### 基本: すべての変更を1コミット、またはファイル単位で1コミット

```bash
git add <file1> <file2>
git commit -m "$(cat <<'EOF'
<type>(<scope>): <要約>

<詳細>
EOF
)"
```

#### 複数コミットに分割する場合（ファイル単位）

変更を複数コミットに分けたいが、各コミットはファイル単位で分割できる場合。
`git stash --keep-index` で、まだコミットしない変更を退避する。

```bash
# 1つ目のコミット
git add <file1>
git stash --keep-index
git commit -m "$(cat <<'EOF'
<type>(<scope>): <要約>

<詳細>
EOF
)"
git stash pop

# 2つ目のコミット
git add <file2>
git commit -m "$(cat <<'EOF'
<type>(<scope>): <要約>

<詳細>
EOF
)"
```

#### 同一ファイル内を hunk 単位で分割する場合

対話的コマンド（`git add -p` / `git add --interactive`）は使用不可。
以下の手順で Edit ツールを使い、最後に検証する。

**前提条件:**
- 新規ファイル（untracked）は hunk 分割の対象外。必ずファイル単位でコミットする。
- 対象は tracked なテキストファイルのみ。binary / symlink は対象外。
- 開始時に clean index であること（staged 変更がない状態）。

**手順:**

```bash
# 1. 検証用スナップショットを保存（pop しない）
git stash
STASH_OID=$(git rev-parse stash@{0})
git stash apply
```

```bash
# 2. 1つ目のコミット用に、含めない hunk を Edit ツールで HEAD の内容へ巻き戻す
#    起点状態: `git stash apply` 直後は全 hunk が作業ツリーに乗っている。
#    操作: このコミットに含めたい hunk はそのまま残し、含めない hunk のみ
#    `git show HEAD:<file>` の対応箇所と一致するように Edit で書き戻す。
git add <file>
git commit -m "$(cat <<'EOF'
<type>(<scope>): <要約>

<詳細>
EOF
)"
```

```bash
# 3. 2つ目のコミット用に、巻き戻していた hunk を Edit ツールで再適用する
#    起点状態: 1 つ目のコミット後は HEAD に含めた hunk のみ反映済みの状態。
#    操作: 残りの hunk を Edit で適用する（stash 内容が既知なので参照しながら）。
git add <file>
git commit -m "$(cat <<'EOF'
<type>(<scope>): <要約>

<詳細>
EOF
)"
```

```bash
# 4. 全コミット完了後、検証
git diff $STASH_OID HEAD
# → 空なら変更漏れなし。差分が出たら漏れまたは余計な変更がある。
```

```bash
# 5. 検証 OK なら stash を破棄
git stash drop
```

**検証が失敗した場合（diff が空でない場合）:**
作業を中断し、`AskUserQuestion` でユーザーに状況を報告する。
勝手に修正を試みない。

## コミットメッセージ形式

```
<type>(<scope>): <日本語の一行要約>

<日本語の詳細説明>
```

### type（英語のまま）

`feat`, `fix`, `docs`, `style`, `refactor`, `perf`, `test`, `build`, `ci`, `chore`, `revert`

#### 曖昧になりやすい典型ケースの指針

複数の type に当てはまりそうな変更でブレないように、以下を優先採用する。

| ケース | 採用 type | 理由 |
|---|---|---|
| 型ヒント・型注釈の追加（挙動不変） | `refactor` | 公開シグネチャの契約変更で、後で `refactor` でも `feat` でも grep されうるが、振る舞い不変なら `refactor` に寄せて履歴の意味を一貫させる |
| f-string 化・文字列結合の書き換え（挙動不変） | `refactor` | 可読性目的のコード書き換え。`style` はフォーマッタ任せの空白/改行調整に限定 |
| import 並び替え・空白/改行調整のみ | `style` | フォーマッタ準拠の見た目だけの変更 |
| 依存パッケージのバージョン更新 | `chore` | ビルド/リリース関連の運用作業。挙動差は `fix` / `feat` の範疇に入れず `chore(deps)` |
| 設定ファイル（lint / CI / editorconfig 等）の調整 | `chore` または `ci` | CI 設定は `ci`、それ以外の運用設定は `chore` |
| テストコードのみ追加・修正 | `test` | 本番コードを触らない場合 |
| ドキュメント（README / コメント / CHANGELOG）のみ | `docs` | 挙動に影響しない記述のみ |
| エラーメッセージ・例外メッセージの改善 | `fix` if 既存バグ起因、`refactor` if 可読性目的 | ユーザー体験影響の有無で振り分け |

### 記載ルール

- type と scope は英語
- subject は日本語で簡潔に一行
- body は日本語で **なぜこの変更が必要だったか** を中心に書く
  - 変更の背景・経緯
  - この実装を選んだ理由・判断根拠
  - 変更ファイルの列挙は不要（diff で分かるため）
- 変更のきっかけ（レビューで指摘された、〜に言われた等）は書かない
  - 誰に言われたかはコードの文脈と無関係
  - 代わりに「〜という問題があった」など、技術的な課題として記述する
  - 例: NG「レビューで〜の指摘を受けた」→ OK「〜という問題があった」
- 変更の背景や意図が不明な場合は `AskUserQuestion` でユーザーに確認してから書く
  - **autonomous / 非対話モード時の例外**: 確認を取れない実行コンテキスト（subagent 実行、自動化スクリプト等）では、コードと diff から **合理的に推測できる動機** を body に記述してよい。その場合は最終応答（または Discretionary fill-ins 相当の箇所）に「推測で書いた」旨を 1 行で明記する
  - **推測の許容範囲**: 以下は OK
    - diff から自明に導けるユースケース（例: `normalize()` で空白を畳み込む → 「後続比較の安定化」）
    - API 設計判断（例: 既存関数の責務を保ったまま別関数化）
    - エラーハンドリングの意図（例: `ValueError` 送出 → 「呼び出し側で捕捉しやすくする」）
  - **推測すべきでないもの**: ビジネス背景の創作（例: 「営業からの要望」「特定顧客の要求」等の外部要因）。誰が・いつ・なぜ依頼したかは diff から導けないので書かない
- subject だけで変更理由が十分伝わる自明な変更（typo 修正、`.gitignore` への追加等）は body を省略してよい
- 1 行あたり 72 文字程度で改行する

## pre-commit hook が失敗した場合

- hook の失敗内容を確認し、問題を修正する
- 修正後は **新しいコミットを作る**（`--amend` は使わない）
  - hook 失敗時はコミットが作成されていないため、`--amend` すると直前の無関係なコミットを書き換えてしまう
- 修正内容が元の変更と論理的に同じなら、1つのコミットにまとめてよい

## 品質チェック

- このコミットは単独で revert 可能か
- 最小の論理単位になっているか
- 変更の背景・意図が body に明確に書かれているか（自明な変更を除く）
- 既存の命名規則と整合しているか

## メッセージの例

```
fix(auth): トークン検証時に例外が適切に処理されない問題を修正

期限切れトークンが送信された場合に 500 エラーが発生していた。
例外処理を追加し、401 を返却するよう修正した。
既存の認証ロジックへの影響はない。
```

```
refactor(search): 検索処理をサービス層へ分離

コントローラに集中していた検索ロジックを SearchService に移動した。
責務を明確化し、テスト容易性を向上させる目的。
挙動自体の変更はない。
```

```
docs(readme): typo を修正
```
